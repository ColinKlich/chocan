Overview

In Project 1, we focused on developing the data processing software for Chocoholics Anonymous. It's important to note that our scope did not encompass the communications software, terminal design, or managing payments between members and providers. Each team member, except for the project submitter, has contributed an HTML file containing their respective use case diagram and team information. This collective effort ensures alignment on the project's content. Erik Iverson is the project submitter. In terms of our glossary, we meticulously defined and clarified terms based on the Chocoholics Anonymous documentation and our Use Case Diagram. The latter was collaboratively created, incorporating input and insights from all group members. You can access the Use Case Diagram by downloading it from the links below or directly from the Team15_Requirements folder submission. The diagram is available in PNG format as "UseCaseDiagram.png," and the diagram file is named "UseCaseDiagram.uxf." Our analysis led us to identify 5 actors and 14 use cases, which were determined through an examination of individual use case diagrams and a comprehensive group assessment. The Use Case diagram is presented as an image, along with a delightful chocolate illustration. For a deeper understanding of our project, we've also provided descriptions of each use case. These descriptions were distributed among the group members to ensure comprehensive coverage. Lastly, we've included a snapshot of our Bitbucket Statistics to showcase everyone's active participation in the project.



Glossary

  • Acme Accounting Services - Responsible for financial procedures such as recording payments of membership fees, suspending members whose fees are overdue, and reinstating suspended members who have now paid what is owing.
  • Automatic Weekly Report Generator - Delivers a report to a provider every Friday at midnight containing a list of services provided to ChocAn members.
  • ChocAn - an organization dedicated to helping people addicted to chocolate.
  • ChocAn Data Center - Handles member and provider verification, as well as billing.
  • EFT - Stands for electronic funds transfer, is the basis for members’ payments.
  • Invalid - A member code is not verified and is invalid because it is not in the system or the member is suspended.
  • Manager - Can access the Manager’s Terminal and request reports and billing information.
  • Manager’s Terminal - What the manager will use to access EFT data, summary reports, and to print summary reports.
  • Member - Pays monthly fee to ChocAn, can receive services pertaining to ChocAn. Each member has their own unique member code.
  • Member Code - A unique nine-digit number that corresponds to the member.
  • Provider - Health care professional that provides services to ChocAn members, and manages the Provider Terminal.
  • Provider Directory - List of service names, codes, and fees.
  • Provider’s Terminal - The terminal that the providers can manage members and the services given to them.
  • Reinstated - When a member resumes payment after being suspended.
  • Service - Consultation or treatment with dietitians, internists, and exercise experts.
  • Service Code - Is a unique sequence of numbers that pertains to a service, such as a consultation or treatment session.
  • Summary report - Weekly report given to a manager. Details providers that need to be paid from a member, number of consultations had by that member, and overall fee to that member. Also details total number of providers that provided services, total number of consultations, and overall fee total.
  • Suspended - When a member has not paid membership fees for at least a month, their account becomes suspended.
  • Validated - A member code is verified and is in the system.

Use Case Descriptions

Use Case: Verify Service Code
Context: The provider inputs their code to gain access to the terminal.
Actors: Provider’s Terminal
Main Success Scenario:
  • 1. The system prompts for a provider code.
  • 2. The system receives a provider code.
  • 3. The system validates the provider code.
  • 4. The system opens up the main menu.
Extensions:
  1. 3a. The provider code was not validated.
  2. 3a.1. The system informs the provider the code was not valid.
  3. 3a.2. The use case continues at step 1.

Use Case: Provide Provider Directory
Context: The provider requests for the Provider Directory containing an alphabetically ordered list of service names and corresponding service codes and fees.
Actors: Provider’s Terminal
Main Success Scenario:
  • 1. The system verifies the service provider code.
  • 2. The system displays the menu on the Provider’s Terminal to select an option.
  • 3. The system displays a button to request the Provider Directory.
  • 4. The system displays what email the Provider Directory will be sent to.
  • 5. The Provider Directory is sent to the provider’s email as an attachment.
Extensions:
  1. 5a. The email did not go through.
  2. 5a.1. The system informs the provider the email did not go through.
  3. 5a.2. The system asks the provider to correct their email.
  4. 5a.3. The use case continues at step 2.

Use Case: Add New Member
Context: The provider adds a new member to the ChocAn Data Center.
Actors: Provider’s Terminal
Main Success Scenario:
  • 1. The system verifies the service provider code.
  • 2. The system displays a menu on the provider terminal to select an option.
  • 3. The provider selects the “Add New Member” option.
  • 4. The system prompts the provider for the member’s information.
  • 4.1 The provider inputs the member’s name into the terminal.
  • 4.2 The provider inputs the member’s street address into the terminal.
  • 4.3 The provider inputs the member’s city into the terminal.
  • 4.4 The provider inputs the member’s state into the terminal
  • 4.5 The provider inputs the member’s ZIP code into the terminal.
  • 5. The provider’s terminal receives the information and stores it into a Member class.
  • 6. The system assigns the new member’s account with a unique member code.
Extensions:
  1. 5a. The address input does not exist
  2. 5a.1 The system informs the Provider Terminal that the information is invalid.
  3. 5b. The member already exists in the system
  4. 5b.1 The system informs the Provider Terminal that the information entered is already associated with a member.
  5. 5b.2. The use case continues at step 4.

Use Case: Verify Member
Context: The terminal then dials the ChocAn Data Center, and verifies the member number.
Actors: Provider’s Terminal, ChocAn Data Center, Acme Accounting Services
Main Success Scenario:
  • 1. Provider slides member’s card through terminal.
  • 2. Terminal dials ChocAn Data Center.
  • 3. ChocAn Data Center verifies number.
  • 4. When number is verified, terminal displays “Validated”.
Extensions:
  1. 4a. If number is invalid, terminal displays reason for invalidity.
  2. 4a.1. The use case continues at step 1.

Use Case: Update Member Records
Context: When run in interactive mode the provider’s terminal allows providers to update member records.
Actors: Provider’s Terminal
Main Success Scenario:
  • 1. The system verifies the service provider code.
  • 2. The system displays the menu on the Provider’s Terminal to select an option.
  • 3. The “Update Member” option is selected.
  • 4. The system prompts for a member code.
  • 5. The system verifies the member code.
  • 6. The system displays the corresponding member’s information.
  • 7. The system allows the provider’s terminal to make and save changes.
  • 7.1 The “save” option is then selected.
  • 7.2 The system receives the updated information.
  • 7.3 The system confirms to the provider’s terminal that the changes have been saved.
  • 8. The system displays the updated member record.
  • 9. The system returns to the main menu.
Extensions:
  1. 5a. The member is suspended.
  2. 5a.1. The system informs the Provider Terminal the member is suspended.
  3. 5a.2. The use case continues at step 2.
  4. 7a. The provider’s terminal has incorrect information (impossible dates/phone numbers etc.)
  5. 7a.1. The system informs the provider’s terminal that the provided information is not correct.
  6. 7a.2. The use case continues at step 7.
  7. 7b. The member’s information could not be saved.
  8. 7b.1 The system informs the provider’s terminal that the updated information could not be saved.
  9. 7b.2 The use case continues at step 7.

Use Case: Delete Member
Context: The provider removes a member from the ChocAn Data Center.
Actors: Provider’s Terminal
Main Success Scenario:
  • 1. The system verifies the service provider code.
  • 2. The system displays a menu on the provider terminal to select an option.
  • 3. The provider selects the “Delete Member” option.
  • 4. The system prompts the provider’s terminal for a member code.
  • 5. The system verifies the member code.
  • 6. The system displays an option on the provider terminal to continue or cancel the use case.
  • 7. The provider selects “Confirm deletion” option.
  • 8. The system clears all stored member information and frees the member code.
Extensions:
  1. 5a. The member code does not exist.
  2. 5a.1. The system informs the provider’s terminal that the member code is invalid.
  3. 5a.2. The use case continues at step 4.

Use Case: Bill Member
Context: The provider records the provision of a service or consultation for a member.
Actors: Provider’s Terminal
Main Success Scenario:
  • 1. The system verifies the service provider code.
  • 2. The provider’s terminal prompts for the member code.
  • 3. The system verifies the member code.
  • 4. The provider inputs the date that service was provided in MM-DD-YYYY format.
  • 5. The provider inputs the six-digit service code into the terminal.
  • 6. The system verifies the service code.
  • 7. The provider’s terminal prompts the provider to confirm the service information.
  • 8. The system writes a record of the service provided to a unique file.
  • 9. The system fetches the fee of the service.
  • 10. The provider’s terminal prompts the provider to confirm the service fee.
  • 11. The system stores the service record and fee.
Extensions:
  1. 4a. The date is invalid.
  2. 4a.1. The provider’s terminal displays an error.
  3. 4a.2. The use case continues at step 4.
  4. 6a. The service code is invalid.
  5. 6a.1. The provider’s terminal displays an error.
  6. 6a.2. The use case continues at step 5.
  7. 7a. The provider chooses to update the service provided.
  8. 7a.1. The use case continues at step 5.
  9. 8a. The provider chooses to update the service record.
  10. 8a.1. The use case continues at step 4.

Use Case: Suspend Member
Context: When a member has not paid membership fees for over a month, their member status becomes suspended, and they cannot receive services until they are reinstated.
Actors: Acme Accounting Services, Provider’s Terminal
Main Success Scenario:
  • 1. Member has a valid account.
  • 2. The member has not paid membership fees for at least a month.
  • 3. Terminal displays “Member suspended".
  • 4. Member removes suspended status when fees are paid.
Extensions:
  1. 2a. Member has paid fees, but system has not verified payment.
  2. 4a. Member paid fees in full, but system does not remove suspended status.

Use Case: Reinstate Member
Context: When a suspended member pays what is owed, they are reinstated, and can resume using the benefits of their membership (services).
Actors: Acme Accounting Services, Provider’s Terminal
Main Success Scenario:
  • 1. Provider’s terminal shows member has a valid account.
  • 2. Member’s account is listed as suspended.
  • 3. Member pays amount owed in full.
  • 4. Acme reinstates member’s account and removes suspended status.
Extensions:
  1. 2a. Member falsely listed as suspended when no fees are owed.
  2. 4a. Provider’s Terminal still lists member as having suspended account.

Use Case: Provide Member Payment Data
Context: Acme Accounting Services is responsible for recording and processing payments of each member.
Actors: Acme Accounting Services
Main Success Scenario:
  • 1. The system displays the list of options for Acme Accounting Services.
  • 2. The system displays the option “Member Payment Information”.
  • 3. All member payment information is displayed.
  • 4. Acme computer automatically updates ChocAn Data Center at 9 P.M. with new payment information for each member
Extensions:
  1. 3a. Member has not paid which will lead to suspension
  2. 4a. Computer fails to automatically update new member payment information and must be manually updated

Use Case: Produce Summary Report
Context: A summary report is given to the manager for accounts payable. The report lists every provider to be paid that week, the number of consultations each had, and their total fee for that week. Finally, the total number of providers who provided services, the total number of consultations, and the overall fee total are printed.
Actors: Manager’s Terminal
Main Success Scenario:
  • 1. The system displays the menu on the Manager’s Terminal to select an option.
  • 2. The system displays that a new summary report is available.
  • 3. The “View Summary Report” is selected.
  • 4. The summary report lists every provider to be paid that week, the number of consultations each had, and his or her total fee for that week. Finally, the total number of providers who provided services, the total number of consultations, and the overall fee total are displayed to the Manager’s Terminal.
Extensions:
  1. 4a. The summary report could not be displayed.
  2. 4a.1 The system informs the Manager’s Terminal that the summary report could not be displayed.
  3. 4a.2. The use case continues at step 1.

Use Case: Run Report Printing
Context: The manager can run and print a report at any time during the week. This report contains a list of services provided to a certain member, sorted in order of when the services were provided.
Actors: Manager’s Terminal
Main Success Scenario:
  • 1. The system displays a list of options the manager can select.
  • 2. The system displays the option “View member reports”.
  • 3. Manager selects desired member.
  • 4. Report is ran displaying the selected member’s information.
Extensions:
  1. 3a. Member report is not available.
  2. 4a. Report displays incorrect information.

Use Case: Run Main Accounting Procedure extends Produce Summary Report and Run Reporting Printing
Context: This system compiles the week’s financial data and creates transaction reports for members, providers, and managers at their request.
Actors: ChocAn Data Center, Automatic Weekly Report Generator
Main Success Scenario:
  • 1. The transactions are recorded in the ChocAn Data Center.
  • 2. The Automatic Weekly Report Generator receives data from the Data Center and creates reports for members and providers of their respective services purchased/provided, along with their account information. It also includes a revenue total to the provider.
  • 3. The reports are emailed to the members and providers.
Extensions:
  1. 1a. If transactions fail to be recorded by the ChocAn data center they could be recovered by the EFT disk.
  2. 3a. Individual reports are provided to managers throughout the week if requested.

Use Case: Providing EFT Data
Context: All electronic funds transfers are recorded to a disk so that they can later validate credits to providers.
Actors: Manager’s Terminal
Main Success Scenario:
  • 1. The system saves transaction data to a disk.
  • 2. A financial summary report is provided weekly to the manager.
  • 3. Banking computers use the data saved in the disk to corroborate provider bank transactions.
Extensions:
  1. 3a. Manager and providers are notified when disk records and bank records do not match.
